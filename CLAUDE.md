# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this repo is

The org-level `.github` repo for the `inference-gateway` GitHub organization. It holds no runtime code — only:

- `profile/README.md` — what GitHub renders on the org page.
- `.github/ISSUE_TEMPLATE/` — org-default templates inherited by any repo without local ones.
- `.github/workflows/` — cross-repo orchestrators that fan out across downstream repos.
- `.github/actions/resolve-targets/` — the shared composite action every orchestrator uses to turn `repos.yaml` + a `select` expression (+ optional `repository`) into a matrix.
- `repos.yaml` — the single registry of downstream targets that drives every orchestrator's matrix.

There is no build, no test suite, no application. Changes here are YAML/Markdown. `README.md` is the authoritative narrative for what each workflow does — read it before editing workflows.

## `repos.yaml` is the single source of truth

Every orchestrator reads `repos.yaml`. Adding or removing a downstream repo = one PR editing this file.

It is **one list** (`targets`); there is no separate `claude_targets`. Each entry has `name`, `kind`, `language`, and an optional nested `orchestrators:` block grouping the per-bot reusable-workflow inputs: `orchestrators.claude` (for repos with a centralized `claude.yml` caller), `orchestrators.infer` (for repos with a centralized `infer.yml` caller), and `orchestrators.codex` (for repos with a centralized `codex.yml` caller). `language` is the single, lowercase toolchain (`go | rust | typescript | python | node | bun`) — it feeds the sync audit-prompt display, the adl `--language` flag in `refresh-agent-manifest.yml`, and (injected by the migrate workflows) the `language` input of each reusable bot workflow. The bot sub-blocks therefore carry only non-language inputs; a repo needing no extras uses an explicit empty map (`claude: {}` / `infer: {}` / `codex: {}`) so the `!= null` selection still matches it.

`kind` values and which workflows pick them up:

| `kind`     | Picked up by                                                |
| ---------- | ----------------------------------------------------------- |
| `sdk`      | `sync-sdks.yml`, `stale.yml`, `cleanup-runs.yml`, `backfill-roadmap.yml`, `migrate-claude.yml`, `migrate-infer.yml`, `migrate-codex.yml` |
| `docs`     | `sync-sdks.yml` (separate drift class E), `stale.yml`, `cleanup-runs.yml`, `backfill-roadmap.yml`, `migrate-claude.yml`, `migrate-infer.yml`, `migrate-codex.yml` |
| `adk`      | `sync-adks.yml`, `stale.yml`, `cleanup-runs.yml`, `backfill-roadmap.yml`, `migrate-claude.yml`, `migrate-infer.yml`, `migrate-codex.yml` |
| `agent`    | `bump-adl.yml`, `refresh-agent-manifest.yml`, `trigger-cd.yml`, `stale.yml`, `cleanup-runs.yml`, `backfill-roadmap.yml` |
| `none`     | `cleanup-runs.yml`, `backfill-roadmap.yml`, `migrate-claude.yml`, `migrate-infer.yml`, `migrate-codex.yml` (infra/tooling repos with no sync/agent family; swept by `cleanup-runs.yml` and `backfill-roadmap.yml` but intentionally **not** by `stale.yml`) |

`migrate-claude.yml` operates on every entry with an `orchestrators.claude` block, `migrate-infer.yml` on every entry with an `orchestrators.infer` block, and `migrate-codex.yml` on every entry with an `orchestrators.codex` block — that is every kind **except** `agent` (agents' `claude.yml` is generated by adl-cli, not migrated). All three bot blocks currently cover the same set of repos.

Target resolution is centralized in the **`.github/actions/resolve-targets`** composite action: a single `yq` read of `.targets`, a jq `select` expression passed per workflow, and an optional `repository` input to narrow to one or more targets (a single name or a comma-separated list, e.g. `docs,infer-action,schemas`; whitespace tolerated, and any requested name absent from the filtered set fails the job listing every miss). The canonical selects:

- `.kind == "agent"` - `bump-adl.yml`, `refresh-agent-manifest.yml`, `trigger-cd.yml`.
- `.kind == "sdk" or .kind == "docs"` - `sync-sdks.yml`.
- `.kind == "adk"` - `sync-adks.yml`.
- `.kind != "none"` - `stale.yml` (every product repo except the `kind: none` infra repos).
- `true` (all entries) - `cleanup-runs.yml` and `backfill-roadmap.yml` (every registered repo, including `kind: none`; private repos are never registered in `repos.yaml`).
- `.orchestrators.claude != null` - `migrate-claude.yml` (every repo with a centralized `claude.yml` caller).
- `.orchestrators.infer != null` - `migrate-infer.yml` (every repo with a centralized `infer.yml` caller).
- `.orchestrators.codex != null` - `migrate-codex.yml` (every repo with a centralized `codex.yml` caller; the same non-agent set as claude/infer).

Add a new workflow by passing the `select` it needs to `resolve-targets`; don't reintroduce per-workflow `yq` filters.

## Three workflow families, different rules

### Sync orchestrators (`sync-sdks.yml`, `sync-adks.yml`) — issues only

These run `anthropics/claude-code-action` with an audit prompt and file structured drift issues. They default `dry_run: true` on manual dispatch (audit + print the would-file list, mutate no issue); the automatic `repository_dispatch` from `schemas` runs for real via `${{ inputs.dry_run || 'false' }}`. The dry-run guard is enforced inside the prompt, since issue filing happens there. Hard invariants when editing their prompts:

- **They inline the org's issue-filing mechanics.** The Issue Type id table and the `updateIssueIssueType` mutation live in each sync prompt (no skill is installed); keep the workflow-specific drift classes, exact titles, labels, and hard safety rules inline here too.
- **Never open PRs, never modify target code, never mention `@claude`.** Issues are notifications for a human reviewer.
- **Issue titles are load-bearing for idempotency.** Re-runs find existing issues via byte-exact title match (`gh issue list --search 'in:title "<exact title>"'`). Changing a title in the prompt orphans every open issue using the old title — open issues will pile up as duplicates. The title→class→Issue Type→labels table in each prompt is the contract.
- **`gh issue create/edit` has no `--type` flag.** Setting the Issue Type requires the GraphQL `updateIssueIssueType` mutation after creation. Each sync prompt inlines that mutation plus the Bug/Feature/Task type-id table; preserve that procedure - including the "Type not in table → warn and leave untyped" rule, which is what keeps class E (`documentation`, no org type) untyped - when editing either side.
- **Proxy endpoints are exempt** from drift classes A, C, E across every SDK and docs target (`proxyGet/Post/Put/Delete/Patch` under `/proxy/{provider}/{path}`). This exemption is explicitly called out as load-bearing because the tracker was repeatedly polluted by false-positive class A issues about these endpoints.
- **`docs` target is ASCII-only.** No em dashes (`—` U+2014) or en dashes (`–` U+2013) anywhere in titles, bodies, or comments — plain hyphen-minus (`-`) only. Applies to every character emitted for that target.
- **Dry-run is prompt-enforced.** When `DRY_RUN` is `true` the prompt forbids `gh issue create/edit/comment/close` and `updateIssueIssueType`; the agent prints the would-file list (byte-exact titles, since titles are idempotency keys) instead. The read-only `gh issue list` idempotency search is still allowed.

### Agent orchestrators (`bump-adl.yml`, `refresh-agent-manifest.yml`, `trigger-cd.yml`) — PRs or dispatch

These open PRs (mechanical, reviewable) or fire `workflow_dispatch` calls. All three are manual-only, default `dry_run: true` (preview the diff/plan, no PR/dispatch), and take an optional `repository` input to run on one agent first. Distinct contracts:

- **`bump-adl.yml` regenerates code and asserts `agent.yaml` is byte-identical afterwards.** Bumping the ADL CLI pin must not change the manifest — if the generator touches `agent.yaml`, the workflow fails by design. The companion workflow is the only legitimate path to modify `agent.yaml`.
- **`refresh-agent-manifest.yml` is the only workflow that edits `agent.yaml`,** and only additively. It snapshots the paths in `PRESERVE_KEYS` (`spec.tools`, `spec.skills`, `spec.config`, `spec.services`), runs an additive deep merge of the fresh `adl init --defaults` scaffold (existing values win), then restores the preserved paths byte-for-byte. Keep the snapshot→merge→restore→validate order intact.
- **`trigger-cd.yml` is fire-and-forget.** It dispatches each agent's own `cd.yml` and exits without waiting. Outcomes are observed on each target's Actions tab.

### Lifecycle orchestrators (`stale.yml`, `cleanup-runs.yml`, `backfill-roadmap.yml`) — cross-repo sweeps across `repos.yaml`

Daily cron + manual `workflow_dispatch`. All three default `dry_run: true` on manual dispatch (preview only) and take an optional `repository` input to sweep one target first; the cron runs for real via `${{ inputs.dry_run || 'false' }}`. `stale.yml` contracts:

- **Cannot use `actions/stale`.** That action is hard-coded to `github.context.repo`; there is no input to target a foreign repo. The orchestrator does the equivalent via `gh issue list --search "updated:<… -label:stale …"` → label + comment, then `gh issue list --label stale --search "updated:<…"` → close.
- **Matrix is `select(.kind != "none")`** — every target except the infra/tooling repos. Those repos (`cli`, `operator`, `registry`, `inference-gateway`, `adl`, …) are `kind: none` and are intentionally not swept; `awesome-a2a` and anything else absent from `repos.yaml` is likewise untouched.
- **Defaults: 30 d → mark `stale` → 7 d → close.** Exempt labels: `pinned`, `security`, `sdk-drift`, `adk-drift`, `docs-coverage`. The drift labels matter - the sync orchestrators (sync-sdks, sync-adks) file `[FEATURE] / [TASK] / [DOCS]` tickets that may legitimately sit open while a maintainer triages, so they must never get auto-closed. (`docs-coverage` is legacy from the removed audit-docs-coverage workflow; kept exempt so any historical tickets survive.)
- **Search-query semantics rely on `updatedAt`.** Adding a label or comment bumps `updatedAt`, so the close step's `updated:<7d ago` filter correctly excludes any issue that received activity (including ours) within the grace window. This is why removal of the `stale` label on re-activity is *not* implemented - the close query already does the right thing.
- **Issues only.** PRs are intentionally untouched.

`cleanup-runs.yml` contracts:

- **Deletes workflow runs, not issues.** Deletes completed runs whose `conclusion` is in the `conclusions` input (default `skipped`; `all` = any conclusion), honoring a `keep_last` retention floor that protects the newest N runs **across the repo** (regardless of workflow). Lists `?status=completed` and ranks/filters in one jq pass (a server-side conclusion pre-filter is incompatible with `keep_last`, which must see every completed run to find the newest N). A per-run re-check re-confirms `status: completed` **and** `conclusion ∈ set` immediately before each `DELETE`, so an in_progress/queued run is never deleted. Defaults (`conclusions=skipped`, `keep_last=0`) reproduce the original skipped-only sweep, so the commented cron stays safe when enabled.
- **`keep_last` protects only *live* workflows.** The step first lists `actions/workflows` (the IDs whose file still exists) and applies `keep_last` only to those; runs of a **deleted/renamed workflow** (a `workflow_id` absent from that set) are pruned in **full**, so a retired pipeline's history doesn't linger forever behind the retention floor.
- **Fail-loud and resumable, not silently green.** Every listing/re-check/delete is rate-limit aware. All matrix jobs share **one** App installation token (a single 5,000-req/hr bucket), so a large `conclusions=all` purge across many repos can exhaust it mid-sweep. A 403 on the run/workflow listing — which previously yielded a false `found 0` and a green check (the process-substitution `mapfile < <(...)` hid the non-zero exit from `set -e`) — now **exits the job non-zero** with an `::error::` and the leftover count. Deletions are idempotent: rerun the workflow (or wait for the daily cron) to continue, and prefer `-f repository=<name>` to purge one repo at a time and stay within the shared budget.
- **Matrix is `select(true)`** — every registered `repos.yaml` target, including the `kind: none` infra repos (`cli`, `operator`, `inference-gateway`, `registry`, ...). This intentionally diverges from `stale.yml` (which exempts `kind: none`): infra repos accumulate skipped `@claude`/`@infer` runs that should be pruned even though their issue trackers should not be auto-staled. Private repos are never registered in `repos.yaml`, so they are excluded by construction; out-of-`repos.yaml` repos (`.github`, `agents`, `awesome-a2a`, `tools`) remain untouched. The optional `repository` input narrows the matrix to one target for testing; narrowing lives in the `resolve-targets` action's jq, not a separate `yq` shape.
- **`actions: write` per target** via the scoped App token (already granted; `trigger-cd.yml` uses the same scope). No `claude-code-action`, no `CLAUDE_CODE_OAUTH_TOKEN`.
- **`dry_run` defaults to `true` on manual dispatch** (deletion is irreversible); the scheduled run deletes. The `schedule:` block ships commented out until validated on one repo. A `MAX_DELETES` per-target cap bounds a single run; hitting it logs a `::warning::` with the leftover count (no silent caps) and the next run continues.

`backfill-roadmap.yml` contracts:

- **Adds issues AND PRs to the board, never deletes or edits them.** For each target it lists **open** issues and **open** PRs that are on **no** project board and adds them to the org Roadmap 2026 board (project #7), then sets their Status to `Todo`. It backfills work that never went through `@claude`/`@infer` (filed by a human, or before board tracking existed) so planned work spread across repos is not lost off the roadmap. The `@claude`/`@infer` bots remain the source of `In progress`/`QA`; this sweep only puts orphans on the board at `Todo` for a human to triage.
- **PRs come from any author; only drafts are skipped.** Open PRs are added regardless of author — humans **and** bots (dependabot, the maintainer App's own release/migrate/bump and `@claude`/`@infer` PRs) — with the single filter `isDraft == false`. Issues are added unfiltered. No per-item Type field is written: the board separates issues from PRs natively via the Projects v2 `is:issue`/`is:pr` view filter (a board-UI concern, not a workflow one). (A PR whose linked issue is already board-tracked can therefore appear as its own `Todo` item; that's accepted - it's still "on no board" by the `projectItems` test.)
- **"On no board" = `projectItems` empty (the load-bearing detection).** The primary filter is the item's ProjectsV2 membership, applied to **both** listings: `gh issue list --json projectItems` and `gh pr list --json projectItems,isDraft`, keeping length 0 (both subcommands expose `projectItems`). The maintainer App's **org-level Projects: read** grant lets `projectItems` reflect membership across **all** org boards (ProjectsV2 connections are viewer-permission-filtered), so an item already on #7 **or any other org board** has length > 0 and is left alone - the semantics are "not on **ANY** project", not "not on #7". This is why there is **no** prefetch of project #7's items (that would answer "not on #7", re-adding items already tracked elsewhere). `projectCards` (classic Projects, shut down 2024-2025) is always empty now and is intentionally **not** checked.
- **Idempotent + resumable.** Issue and PR URLs are merged into one add/Status loop (`item-add --url` accepts either, and `num="${url##*/}"` extracts the number from `/issues/N` or `/pull/N` alike — issue and PR numbers share one per-repo space, so no collision). `gh project item-add 7` on an item already on #7 just returns the existing item id (exit 0), so re-runs are no-ops and the `projectItems` filter shrinks the candidate set each run - a partial/capped run is safe to rerun. A `MAX_ADDS` per-target cap bounds a single run; hitting it logs a `::warning::` with the leftover count (no silent caps) and the next run continues.
- **Matrix is `select(true)`** — every registered `repos.yaml` target, including `kind: none` infra and `kind: agent` repos (orphan issues and PRs can live anywhere). Like `cleanup-runs.yml` and unlike `stale.yml` (which exempts `kind: none`). Private repos and out-of-`repos.yaml` repos (`.github`, `agents`, `awesome-a2a`, `tools`) are untouched by construction. Open issues and open PRs only (`--state open`).
- **Auth: per-target scoped App token; the org Projects R/W grant rides along despite repo-scoping** (org permissions are not narrowed by `repositories:`), exactly how `claude.yml`/`infer.yml` write to #7 from repo-scoped tokens. Listing uses the App's installation-wide `issues: read` plus `pull-requests: read` (subsumed by the `pull-requests: write` the bots already hold to open PRs — no new scope). No `claude-code-action`, no `CLAUDE_CODE_OAUTH_TOKEN`. The board project/field/option ids are org-stable and hard-coded here (the same ids are inlined in `claude.yml`/`infer.yml`); re-derive only if a command rejects one.
- **Fail-loud and resumable, not silently green.** Like `cleanup-runs.yml`, a 403 on **either** the issue listing or the PR listing **exits the job non-zero** with an `::error::` and leftover count rather than the process-substitution `mapfile < <(...)` hiding the failure as a false `found 0`. `dry_run` defaults to `true` on manual dispatch; the `schedule:` block (cron `30 4 * * *`, staggered off `stale.yml` and `cleanup-runs.yml`) ships commented out until validated on one repo.

### Reusable bot workflows (`claude.yml`, `infer.yml`) + their migrations

Three reusable workflows (`on: workflow_call`) centralize the org's AI bots so each per-repo caller is ~15 lines: `claude.yml` wraps `anthropics/claude-code-action` (`@claude`); `infer.yml` wraps `inference-gateway/infer-action@v<version>` (`@infer`, default model `ollama_cloud/deepseek-v4-flash`); `codex.yml` wraps `openai/codex-action@v1.8` (`@codex`). All pin their action to an exact tag and cap the job at `timeout-minutes: 25`. The `on:` triggers and the `@claude`/`@infer`/`@codex` gate `if:` live in each thin caller, not in the reusable workflow.

All three share a `debug` input (default `false`, settable per-repo via the entry's `orchestrators.<bot>.debug` in `repos.yaml`) that surfaces verbose agent logs in the Actions run log - intended for **private** repos only, since debug/transcript output can include sensitive values. Each bot maps it to its action's own knobs: `claude.yml` appends `--debug` to `claude_args` (stderr) and sets `show_full_output`; `infer.yml` sets both infer-action `debug` **and** `mirror-agent-logs` (the agent stdout transcript) from the one flag; `codex.yml` passes `codex-args: --json` (full event transcript on stdout) and `RUST_LOG=debug` (internal tracing on stderr). Because the migrate workflows pass every `orchestrators.<bot>` sub-key straight through, flipping `debug: true` needs no migrate-workflow change - only a re-migrate of the target against a `.github` ref that includes the input (claude/codex are new; infer's `debug` predates this but its `mirror-agent-logs` coupling is new).

On `issues`/`issue_comment` events, `claude.yml` also tracks the worked issue on the org Roadmap 2026 board (project #7): it adds the issue and sets Status `In progress` before implementing, then `QA` when it opens the PR. It does **not** set `Done` - that is a later, post-merge transition owned outside the run (today a maintainer; a future merge-triggered automation could own it). Board mechanics (project/field/option ids, the `gh project` `item-add`/`item-list`/`item-edit` commands), the docs + downstream-impact follow-up rules, the no-em/en-dash writing style, and the permission-escalation rule all come from the shared **`bot-instructions` composite action** (`.github/actions/bot-instructions`, outputs `board`/`followups`/`style`/`releases`/`sandbox`/`escalation`/`combined` from the matching `.md` files), injected into `claude.yml`'s prompt and `infer.yml`'s `custom-instructions` so they are maintained once for both bots; referenced cross-repo at `@main`, so editing them propagates org-wide. The agent's `gh` authenticates as the maintainer App because `github_token` is the App token and claude-code-action exports it to the agent as `GH_TOKEN`/`GITHUB_TOKEN` (run.ts), so no extra step-level token env is needed; the App's **org-level Projects: read & write** is what actually reaches project #7 (there is no `projects:` key in a workflow `permissions:` block), and its owner-wide install is what lets it file into `.github`. Any permission failure in any context (the rule lives in the shared prompt block, so it also covers PR/CI-fix and manual runs) triggers a best-effort, idempotent escalation issue in `inference-gateway/.github` (title `[bot] Missing GitHub permission: <capability>`, label `bot`, deduped org-wide so many repos collapse to one issue + comments) that must never abort the real task and never mention `@claude`. Board membership is orthogonal to `track_progress` (the action's own progress comment) - do not conflate the two.

`claude.yml` resolves a per-run **browser mode** in a "Resolve browser mode" step (output `steps.browser.outputs.enabled`) = OR of: the static `browser` input (set per-repo in `repos.yaml`, e.g. `adl`), the `browser` dispatch checkbox (forwarded by the thin caller as `browser: ${{ inputs.browser || <static> }}`), and a case-insensitive literal `[browser]` marker in the issue/comment/review body or issue title (read via `env:` for injection safety, matched with `grep -qiF` since `[browser]` is otherwise a glob character class). When enabled it gates the Node setup + "Install Playwright + Chromium" steps and adds the `@playwright/mcp` server + `mcp__playwright` tool to `--mcp-config`/`--allowedTools` for that run (the MCP server/tool keep the `playwright` name; only the config input is `browser`), and appends a one-line browser-available note to the prompt. Thin callers reference the reusable workflow at `@main` (the migrate workflows' default; a `ref` input can still pin a tag), so changes here reach callers on merge - no release + re-migrate rollout.

`claude.yml` persists Claude Code **auto-memory** across runs by syncing the private `inference-gateway/.memory` repo's dedicated **`claude` branch** (gated by a `memory` input, default `true`, per-repo opt-out via `orchestrators.claude.memory: false`). `@infer` owns `.memory`'s `main` branch and the two agents' on-disk memory layouts differ, so they are isolated on separate branches - clobbering *within* claude's own branch across runs is expected (it is one org-wide pool, like infer's). A "Sync memory in" step (before "Run Claude Code", `if: inputs.memory`) clones `.memory@claude` into **`~/.claude/memory`** (Claude's home dir; falling back to a fresh `git init -b claude` when the branch does not exist yet) and emits `{"autoMemoryDirectory":"$HOME/.claude/memory"}` as claude-code-action's `settings` input - pinning Claude's memory to that fixed path instead of its default project-scoped `~/.claude/projects/<slug>/memory/` (which is machine-local, wiped each fresh runner, and an awkward slug to target for git sync). `~/.claude/memory` lives in `$HOME`, **outside `$GITHUB_WORKSPACE`**, so neither the claude-code-action commit nor the "Preserve work-in-progress" `git add -A` (both operate on the workspace) can leak memory files into the target repo's PR. A "Sync memory out" step (`if: ${{ always() && inputs.memory }}`, so it runs even on failure/cancel) `git add -A`s and, when nothing changed, no-ops with a `$GITHUB_STEP_SUMMARY` line; otherwise it commits `chore(memory): sync ...` and **fast-forward** pushes `HEAD:claude` (a concurrent run that loses the race is rejected and warns, never force-clobbers). It is best-effort throughout (`exit 0` on every path - a memory hiccup never reddens an otherwise-green Claude run), mirroring infer's best-effort backend. `.memory` is in the bot App tokens' `repositories:` allow-list (the App has `contents: write` there). Callers at `@main` pick this up on merge; and the `kind: agent` repos generate their `claude.yml` from adl-cli, not this reusable workflow, so they do not get memory from here.

`infer.yml` now carries the same two behaviors, injected through infer-action's `custom-instructions` input (which **appends** to the action's built-in plan/progress/PR workflow rather than overriding a `system-prompt-*`): on an issue trigger the `@infer` agent adds the worked issue to project #7 and advances Status `In progress -> QA` (never `Done`), and on a genuine permission failure it files the same idempotent `[bot] Missing GitHub permission: <capability>` escalation issue (label `bot`) in `.github`. Two infer-specific wrinkles vs. `claude.yml`: the agent resolves the issue URL from its own task context (there is no workflow-side `__URL__`/`__N__` substitution). All three bot workflows mint the App token with an explicit `repositories:` allow-list of **every public repo registered in `repos.yaml`** (plus `.github`, and `.memory` for claude/infer) so cross-repo follow-up issues can be created *and labelled* on any registered repo (a token narrowed to just the current repo could still create issues on public repos but not label them - see `.github#93`); the list is static, so registering a new repo in `repos.yaml` means also adding it to the three workflows' `repositories:` blocks. Org project #7 rides the installation-level Projects grant either way. Because both reusable bots now mint from the same `INFERENCE_GATEWAY_MAINTAINER_APP_*` maintainer App, the single org Projects R/W grant covers `@infer` too - no separate App or grant. The board/issue write tools (`gh project item-add/item-edit`, `gh issue create/edit/comment`, plus the tightly scoped `updateIssueIssueType` graphql) are appended **unconditionally** by infer.yml's bash-allow-append step (no longer gated on `skills: maintainer`, which now defaults empty), and the `custom-instructions` text is composed **per trigger** from the shared `bot-instructions` action's individual outputs (the same copy `claude.yml` injects), mirroring `claude.yml`'s "Compose prompt" context resolution: issue-triggered runs (`issues`, or `issue_comment` on a non-PR issue) get the full `board`/`followups`/`style`/`releases`/`sandbox`/`escalation` set; PR-comment and `workflow_dispatch` runs drop `board` (~350 tokens that only apply to issue work), with `escalation` always last.

`infer.yml` also resolves a per-run **Claude Code subscription mode** in a "Resolve subscription mode" step (output `steps.subscription.outputs.enabled` / `.model`), mirroring `claude.yml`'s `[browser]` resolution pattern: it scans `inputs.prompt` (the `workflow_dispatch` free-text) + the issue title/body + comment/review body (injected via `env:` for safety) for a case-insensitive `/subscription anthropic/<model>` directive. When present, it sets `enabled=true`, extracts the bare Claude id (stripping the `anthropic/` prefix, since infer-action's subscription mode requires a bare id like `claude-sonnet-4-6`), and the "Run Infer Agent" step passes `use-claude-code-subscription: true`, `claude-code-oauth-token: ${{ secrets.CLAUDE_CODE_OAUTH_TOKEN }}`, and the resolved bare model to infer-action instead of the default `ANTHROPIC_API_KEY` + `inputs.model`. `CLAUDE_CODE_OAUTH_TOKEN` is declared as an optional (`required: false`) `workflow_call` secret on `infer.yml`; callers already use `secrets: inherit`, so no per-repo caller or `migrate-infer.yml` change is needed (callers at `@main` pick it up on merge). When the directive is absent, `enabled=false` / `model=""`, so `model` resolves to `inputs.model` and the OAuth input is empty - default behavior is byte-for-byte unchanged.

`codex.yml` adds a third bot, `@codex`, with a different internal shape because `openai/codex-action` is a **bare `codex exec` runner**: it edits the working tree and returns a `final-message`, but does **not** commit, push, open PRs, post comments, or hand the model a `GH_TOKEN`. So unlike `claude.yml`/`infer.yml` (which delegate the whole issue->PR flow to their actions), `codex.yml` performs every GitHub side-effect as explicit workflow steps with the maintainer App token: it sets the worked issue's board Status to `In progress` (deterministic `gh project` steps, **not** the agent, since codex has no `gh`), assembles the task + the shared `bot-instructions` `style` block into a `prompt-file`, runs codex (`sandbox: workspace-write` - edits the tree and reaches the OpenAI API), then opens the PR via `peter-evans/create-pull-request`, posts `final-message` as a PR comment (or, when codex made no changes, an issue comment), and advances Status to `QA`. The board project/field/option ids are inlined verbatim from `bot-instructions/board.md` (codex cannot read them from the agent-prompt outputs since it does not run `gh`). Its caller carries the same bot-author loop filter as `infer.yml`'s (the App opens the PR as `…[bot]`) and restricts `issue_comment` to non-PR issues — PR-review feedback is out of scope (codex-action has no native PR-branch iteration). Needs `OPENAI_API_KEY` (already an org secret for `infer.yml`) plus the maintainer App, both via `secrets: inherit`.

`migrate-claude.yml` / `migrate-infer.yml` are one-shot fan-outs that write the thin caller into each target and open a PR. They select `.orchestrators.claude != null` / `.orchestrators.infer != null` and build the caller's `with:` inputs by prepending the entry's top-level `language` to that repo's `orchestrators.claude` / `orchestrators.infer` block. `migrate-infer.yml` additionally bumps the **target's own** Flox `infer` pin (`.flox/env/manifest.toml`) to the latest `inference-gateway/cli` release and refreshes the lock in the **same PR**. It mirrors `migrate-claude.yml`: detect the target's current pin, `sed` it to latest, `flox upgrade infer`, then a bare `flox activate -- true` to write `manifest.lock` in its canonical form, then commit `manifest.toml` + `manifest.lock` + `infer.yml` together. No per-run version input - the matrix job resolves the latest release via `gh api repos/inference-gateway/cli/releases/latest`. `migrate-claude.yml` does the same flox bump for its bot's CLI: it bumps the **target's own** Flox `claude-code` pin (`.flox/env/manifest.toml`) to the latest version and refreshes `manifest.lock`, committing `claude.yml` + `manifest.toml` + `manifest.lock` together. Two distribution-model differences from infer: `claude-code` is a flox **catalog** package (`claude-code.version = "^X.Y.Z"`, a caret floor), **not** a GitHub-release flake - so the latest comes from `flox show claude-code` (NOT `gh api .../releases/latest`; npm's latest can be ahead of the catalog, so never read npm), and the bump `sed`s the caret floor, **isolates claude-code into its own flox `pkg-group` (`"claude-code"`)** so the raised floor resolves independently of co-resident packages (flox resolves each group against one catalog page; the default `ai` group shared with `codex` made the constraints unsatisfiable), then `flox upgrade claude-code` followed by `flox activate -- true`; there is **no** config-regen step (claude-code has no `init` analog), so the activate here is a bare no-op whose only job is to write `manifest.lock` in its canonical form (the `flox upgrade` alone leaves a non-canonical lock) - the same bare `flox activate -- true` migrate-infer now uses. Its PR title names the moves (`chore(deps): bump claude-code X -> Y, claude-code-action vA -> vB`, claude-code-action read from the reusable `claude.yml` at the caller's old vs. new ref), falling back to `ci: centralize claude.yml via reusable workflow`. The generated caller also carries a `browser` boolean dispatch checkbox; because the reusable `browser` input is boolean-typed (and `inputs.browser` is null on non-dispatch events), the render `del(.browser)`s the static jq `with:` block and emits one merged `browser: ${{ inputs.browser || <static> }}` line - the `|| <static>` both coerces a real boolean on issue/comment runs (a bare null would fail the boolean-typed input) and preserves a repo's `repos.yaml` `browser: true`. Both migrations tolerate a target with **no** Flox env: when `.flox/env/manifest.toml` is absent the bump step `flox init`s a fresh env, writes the canonical pin (infer's `infer.flake = "github:inference-gateway/cli/vX.Y.Z"` / claude-code's catalog `pkg-path` + `^version` + own `pkg-group`), locks it with `flox activate -- true`, sets `old=NEW` (so the PR title falls back to the centralize form), and the PR commits the whole `.flox/` dir (`git add .flox`, gitignore-filtered) - so the migration installs the thin caller instead of aborting on the missing manifest. A manifest that exists but lacks the bot's pin stays a hard `exit 1` (a misconfigured registered repo, not an un-floxed one). The two bots coexist (both available per repo); fixing a bot is one PR in `.github` + a tag move.

`migrate-codex.yml` is `@codex`'s migration, converted from the former standalone `bump-codex.yml`: it now writes the thin `codex.yml` caller **and** folds in the Flox `codex` catalog-pin bump in one PR, exactly as `migrate-claude.yml` folds in the `claude-code` bump. It selects `.orchestrators.codex != null`, prepends the entry's top-level `language` to the `orchestrators.codex` block to build the caller's `with:`, `sed`s the `codex.version` caret floor to the latest `flox show codex`, **isolates codex into its own `pkg-group = "codex"`** so the raised floor resolves independently of co-resident packages, then `flox upgrade codex` + bare `flox activate -- true` to canonicalize `manifest.lock`. It bootstraps a Flox env when `.flox/env/manifest.toml` is absent (like the other two migrations); a manifest that exists but lacks a `codex.version` pin stays a hard `exit 1`. PR title `chore(deps): bump codex X -> Y[, codex-action vA -> vB]`, falling back to `ci(codex): centralize codex.yml via reusable workflow` (codex-action ships **no** semver release - only tags `v1`..`v1.8` - so the codex-action version line is best-effort). Needs `contents: write` + `pull-requests: write` + `workflows: write` (it writes `.github/workflows/codex.yml`) - already granted, the same scope `migrate-claude`/`migrate-infer` use; an optional `codex_version` input pins a specific catalog version instead of latest.

## Auth model

The fan-out orchestrators use a single GitHub App, `inference-gateway-maintainer-bot`, to mint per-target scoped tokens via `actions/create-github-app-token` in every job. The reusable `@claude` and `@infer` workflows both mint a maintainer-bot App token (the `INFERENCE_GATEWAY_MAINTAINER_APP_*` secret pair) that the agent's own `gh` then runs as. No PATs anywhere. Required secrets:

- `INFERENCE_GATEWAY_MAINTAINER_APP_ID` / `INFERENCE_GATEWAY_MAINTAINER_APP_PRIVATE_KEY` — maintainer App client ID + private key (every fan-out + migration workflow)
- `CLAUDE_CODE_OAUTH_TOKEN` — for `claude-code-action` (reusable `claude.yml` + the sync workflows)
- `OPENAI_API_KEY` — for `openai/codex-action` (reusable `codex.yml`); already provisioned as an `infer.yml` provider key, now also forwarded to `@codex` via `secrets: inherit`
- `INFERENCE_GATEWAY_MAINTAINER_APP_ID` / `INFERENCE_GATEWAY_MAINTAINER_APP_PRIVATE_KEY` — the maintainer-bot App token minted inside the reusable `claude.yml` and `infer.yml` (the identity the agent's `gh` uses); needs **org-level Projects: read & write** + `issues: write` on `.github` for board tracking and permission escalation
- Provider API keys for `infer.yml` (model-dependent; `DEEPSEEK_API_KEY` covers the default model): `ANTHROPIC_API_KEY`, `OPENAI_API_KEY`, `GOOGLE_API_KEY`, `DEEPSEEK_API_KEY`, `GROQ_API_KEY`, `MISTRAL_API_KEY`, `CLOUDFLARE_API_KEY`, `COHERE_API_KEY`, `OLLAMA_API_KEY`, `OLLAMA_CLOUD_API_KEY`, `MOONSHOT_API_KEY`

Per-workflow App permission requirements are documented in `README.md` under "Pre-merge requirements" — consult it before adding a new orchestrator that needs new scopes.

## Common operations

Every fan-out workflow defaults to `dry_run: true` on manual dispatch (safe preview). Add `-f dry_run=false` to act for real, and `-f repository=<name>` to run on a single target first (or `-f repository=<name1>,<name2>` for a comma-separated subset).

```sh
# Sync (audit + print would-file issues; spec-updated events from `schemas` file for real):
gh workflow run sync-sdks.yml --repo inference-gateway/.github                                            # dry, all sdk/docs
gh workflow run sync-sdks.yml --repo inference-gateway/.github -f dry_run=false -f repository=python-sdk  # file for real, one repo
gh workflow run sync-adks.yml --repo inference-gateway/.github

# Agent fan-out:
gh workflow run bump-adl.yml --repo inference-gateway/.github -f adl_version=vX.Y.Z                       # dry preview
gh workflow run bump-adl.yml --repo inference-gateway/.github -f adl_version=vX.Y.Z -f dry_run=false      # open PRs
gh workflow run refresh-agent-manifest.yml --repo inference-gateway/.github -f adl_version=vX.Y.Z -f repository=mock-agent
gh workflow run trigger-cd.yml --repo inference-gateway/.github -f dry_run=false                          # dispatch cd.yml on all agents

# Migration (one-shot): write a repo's claude.yml / infer.yml thin caller of the reusable workflow:
gh workflow run migrate-claude.yml --repo inference-gateway/.github -f repository=cli                     # dry, one repo
gh workflow run migrate-claude.yml --repo inference-gateway/.github -f dry_run=false                      # open PRs for all
gh workflow run migrate-infer.yml --repo inference-gateway/.github -f repository=cli                      # dry, one repo
gh workflow run migrate-infer.yml --repo inference-gateway/.github -f dry_run=false                       # open PRs for all
# (bumps each target's own .flox/env/manifest.toml infer pin to the latest inference-gateway/cli release)

# Migration (one-shot): write a repo's codex.yml thin caller of the reusable workflow + bump its Flox codex pin:
gh workflow run migrate-codex.yml --repo inference-gateway/.github -f repository=cli                      # dry, one repo
gh workflow run migrate-codex.yml --repo inference-gateway/.github -f dry_run=false                       # open PRs for all
# (writes codex.yml thin caller + bumps each target's own .flox/env/manifest.toml codex pin to the latest catalog version)

# Lifecycle (cron runs for real; manual previews by default):
gh workflow run stale.yml --repo inference-gateway/.github -f dry_run=false                               # sweep for real
gh workflow run cleanup-runs.yml --repo inference-gateway/.github                                               # dry, all targets, skipped-only
gh workflow run cleanup-runs.yml --repo inference-gateway/.github -f dry_run=false -f repository=docs           # delete skipped on one target
gh workflow run cleanup-runs.yml --repo inference-gateway/.github -f conclusions=skipped,failure -f keep_last=5 # dry: prune skipped+failed, keep newest 5 repo-wide
gh workflow run cleanup-runs.yml --repo inference-gateway/.github -f conclusions=all -f keep_last=5             # dry: keep only newest 5 in the repo, prune the rest
gh workflow run backfill-roadmap.yml --repo inference-gateway/.github -f repository=cli                         # dry, one repo: list orphan open issues + PRs
gh workflow run backfill-roadmap.yml --repo inference-gateway/.github                                           # dry, all targets
gh workflow run backfill-roadmap.yml --repo inference-gateway/.github -f dry_run=false -f repository=cli        # add one repo's orphans to project #7 (Status=Todo)
gh workflow run backfill-roadmap.yml --repo inference-gateway/.github -f dry_run=false                          # backfill all targets for real

# Validate repos.yaml shape after edits:
yq '.targets[] | .kind' repos.yaml | sort -u       # should only print: adk agent docs none sdk tooling
yq '.targets[] | .language' repos.yaml | sort -u   # should only print lowercase toolchains: bun go node python rust typescript
yq '.targets[] | select(.orchestrators.claude.language != null or .orchestrators.infer.language != null or .orchestrators.codex.language != null) | .name' repos.yaml  # must be EMPTY: language is top-level only, never a bot sub-key
```

The sync workflows are normally driven by `repository_dispatch` from `inference-gateway/schemas` (`event_type: spec-updated` for OpenAPI, `a2a-spec-updated` for the A2A schema). `workflow_dispatch` is the manual path for testing.

## Local dev environment

A Flox env (`.flox/env/manifest.toml`) provides AI CLIs used during local authoring — `infer` (from `inference-gateway/cli`), `claude-code`, `codex`. These are tools for editing this repo, not dependencies of any workflow. The workflows themselves install their tools (`yq`, `gh`, `task`, `flox`) inside the runner.
